② postgresql.conf（重点）

```conf
shared_preload_libraries = 'citus'
listen_addresses = '*'
```

------

③ pg_hba.conf（非常重要）

允许 Coordinator IP 访问 Worker：

```conf
host  all  all  <COORDINATOR_IP>/32  md5
```

📌 这是 worker 能“被调度”的前提

------

④ 重启 PostgreSQL

```bash
sudo systemctl restart postgresql
```

------

###### 3️⃣ Worker 不需要做的事（新手常误解）

❌ 不需要：

- 手动建分布式表
- 手动建 shard
- 连接应用
- 配任何“主从”

👉 Worker 完全被 Coordinator 控制

------

##### 三、Worker 是怎么和 Coordinator 通信的？（重点）

###### 1️⃣ 通信本质：PostgreSQL 普通连接

> 没有 RPC、没有 gRPC、没有消息队列

就是：

```text
Coordinator
   |
   |  libpq (TCP)
   |
Worker
```

使用的协议：

- PostgreSQL wire protocol
- TCP 5432

------

###### 2️⃣ 通信是谁主动的？

✅ 永远是 Coordinator 主动

```
Client → Coordinator
Coordinator → Worker
```

📌 Worker 不会主动找 Coordinator

------

###### 3️⃣ Coordinator 如何“认识” Worker？

通过这条命令 👇

```sql
SELECT citus_add_node('worker_ip', 5432);
```

这一步做了什么？

- 在 Coordinator 元数据表里记录：
  - worker IP
  - 端口
- 之后所有路由都基于这个信息

------

###### 4️⃣ SQL 执行时的真实通信流程（你一定要看）

查询：

```sql
SELECT count(*)
FROM system_log
WHERE tenant_id = 3;
```

内部流程：

```
1. Coordinator 解析 SQL
2. 找到 tenant_id = 3 → shard_102015
3. shard_102015 在 Worker B
4. Coordinator 建立 PG 连接
5. 发送子 SQL
6. Worker 执行
7. 返回结果
```

📌 这就是“计算分散”的核心

------

###### 5️⃣ INSERT 的通信流程（很轻）

```sql
INSERT INTO system_log VALUES (3, 'xxx');
Coordinator → Worker（1 次网络）
Worker → Coordinator（OK）
```

------

##### 四、部署时你最容易踩的 5 个坑（非常实战）

###### ❌ 1️⃣ Coordinator 和 Worker 版本不一致

- PG / Citus 版本必须一致

------

###### ❌ 2️⃣ pg_hba.conf 没放行

- 这是最常见错误
- 表现为：add_node 成功，执行 SQL 失败

------

###### ❌ 3️⃣ Coordinator 放了大量数据

- JOIN / 查询压力会反噬 coordinator

------

###### ❌ 4️⃣ 网络延迟高

- Citus 对网络非常敏感
- 不要跨机房

------

###### ❌ 5️⃣ 把 Worker 暴露给业务

- 应用只连 coordinator

------

在 Coordinator 执行：

```sql

```

再执行：


##### 六、工程级一句话总结（记住这句）

> Coordinator 不干活，只指挥；Worker 不思考，只执行；它们用的只是最普通的 PostgreSQL 连接。

##### 四、Worker 节点需要做什么？

👉 答案：几乎什么都不用做

- 不用建库（citus 会同步）
- 不用建表
- 不用连应用

只要：

- PostgreSQL 在跑
- 能被 coordinator 访问

你可以在任意 worker 上验证：

```bash

SELECT citus_is_coordinator();
```

👉 返回 f，说明它是 worker

##### 五、验证集群是否真的“分布式”

###### 1️⃣ 在 Coordinator 建表

```sql
CREATE TABLE system_log (
    id BIGSERIAL,
    tenant_id INT,

    created_at TIMESTAMPTZ DEFAULT now()
```

###### 2️⃣ 转成分布式表

##### 六、常见错误 & 对应排查（你很可能会遇到）

###### ❌ add_node 时报错连接失败

👉 99% 是：

- pg_hba.conf 没放行
- 防火墙没开 5432

检查：

```bash
telnet 192.168.1.113 5432
```

------

###### ❌ EXPLAIN 没有 Citus Adaptive

👉 表不是 distributed table

```sql
SELECT * FROM citus_tables;
```

------

###### ❌ 插入很慢

👉 分片键没写 / 值不均匀

##### 七、非常重要的生产建议（先记住）

- ❌ 不要让业务连 worker
- ❌ 不要在 worker 手动建表
- ✅ coordinator 独立一台
- ✅ 所有机器时间同步（NTP）
