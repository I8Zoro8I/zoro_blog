#### pg_trgm —— 模糊搜索 / like / 搜索性能质变

###### 1. 它解决什么问题？

你现在是不是经常写这种 SQL 👇：

```sql
WHERE title ILIKE '%虾%'
   OR desc ILIKE '%虾%'
```

问题是：

- %xxx% 必定全表扫描
- 数据一多直接炸

pg_trgm 能让 模糊搜索走索引。

------

###### 2. 安装插件（只需一次）

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
```

验证：

```sql
SELECT * FROM pg_extension WHERE extname = 'pg_trgm';
```

------

###### 3. 给字段建 trigram 索引（核心）

```sql
CREATE INDEX idx_recipe_title_trgm
ON recipe
USING gin (title gin_trgm_ops);
```

如果你经常：

```sql
title + desc 一起搜
```

可以建 组合索引：

```sql
CREATE INDEX idx_recipe_search_trgm
ON recipe
USING gin ((title || ' ' || desc) gin_trgm_ops);
```

💡 这个在 菜谱 / 日志 / 内容搜索里非常好用。

------

###### 4. 实测效果（你一定要做）

```sql
EXPLAIN ANALYZE
SELECT *
FROM recipe
WHERE title ILIKE '%大虾%';
```

你会看到：

- ❌ 没插件 → Seq Scan
- ✅ 有插件 → Bitmap Index Scan / Index Scan

这一步一定要亲眼看，你才会真的理解插件的价值。

------

###### 5. 你现在项目里可以立刻用的地方

结合你已有的模块 👇

- 菜谱搜索（title / desc）
- 商品名称搜索
- 日志 message 模糊匹配
- 用户备注、操作说明

#### 全文检索 tsvector

tsvector 是 PostgreSQL 内置类型，只要数据库版本 ≥ 8.3（你现在用的肯定远高于这个），开箱即用。

- tsvector：不用安装
- ✅ to_tsvector() / to_tsquery()：直接可用
- ❌ 只有 分词规则 / 扩展能力 才可能需要额外插件

##### 1.日志表结构demo

```sql
CREATE TABLE system_log (
    id          BIGSERIAL PRIMARY KEY,
    log_time    TIMESTAMPTZ NOT NULL,
    level       TEXT,                 -- info / warn / error
    module      TEXT,                 -- order / payment / recipe
    message     TEXT,                 -- 原始日志
    extra       JSONB                 -- 扩展信息
);
```

##### 2.加全文检索列（核心）

```sql
ALTER TABLE system_log
ADD COLUMN search_vec tsvector;
```

##### 3.填充tsvetor

```sql
UPDATE system_log
SET search_vec =
    to_tsvector(
        'simple', --给“机器日志”用的全文检索配置
        coalesce(level,'')   || ' ' ||
        coalesce(module,'')  || ' ' ||
        coalesce(message,'')
    );--手动步骤  正式自动添加
```

注意点：

- 不要只索引 message
- 把 level / module 一起塞进去
- 搜索体验会好很多

##### 4.建 GIN 索引

```sql
CREATE INDEX idx_system_log_search
ON system_log
USING gin (search_vec);
SELECT *
FROM system_log
WHERE search_vec @@ plainto_tsquery('simple', 'error redis timeout');
--等价于：
--包含 error AND redis AND timeout
SELECT *
FROM system_log
WHERE log_time BETWEEN '2025-01-01' AND '2025-01-02'
  AND search_vec @@ websearch_to_tsquery('simple', 'error timeout')
ORDER BY log_time DESC
LIMIT 100;
```

- 时间范围走 B-tree
- 文本走 GIN
- 双索引组合，非常稳

##### 5.权重 & 排序

不同字段不同权重

```sql
UPDATE system_log
SET search_vec =
      setweight(to_tsvector('simple', coalesce(level,'')), 'A')
  ||  setweight(to_tsvector('simple', coalesce(module,'')), 'B')
  ||  setweight(to_tsvector('simple', coalesce(message,'')), 'C');
```

- A > B > C
- 搜索 “error” 时，level=error 的会排前面

按相关度排序

```sql
SELECT *,
       ts_rank(search_vec,
               websearch_to_tsquery('simple', 'error timeout')) AS rank
FROM system_log
WHERE search_vec @@ websearch_to_tsquery('simple', 'error timeout')
ORDER BY rank DESC, log_time DESC
LIMIT 50;
```

非常像 Elasticsearch 的体验

##### 6.自动维护

1-触发器

```sql
CREATE FUNCTION system_log_tsvector_trigger()
RETURNS trigger AS $$
BEGIN
  NEW.search_vec :=
      setweight(to_tsvector('simple', coalesce(NEW.level,'')), 'A')
  ||  setweight(to_tsvector('simple', coalesce(NEW.module,'')), 'B')
  ||  setweight(to_tsvector('simple', coalesce(NEW.message,'')), 'C');
  RETURN NEW;
END
$$ LANGUAGE plpgsql;
CREATE TRIGGER trg_system_log_tsv
BEFORE INSERT OR UPDATE
ON system_log
FOR EACH ROW
EXECUTE FUNCTION system_log_tsvector_trigger();
```

2-生成列(pg12+有这个功能)

```sql
ALTER TABLE system_log
ADD COLUMN search_vec tsvector
GENERATED ALWAYS AS (
      setweight(to_tsvector('simple', coalesce(level,'')), 'A')
  ||  setweight(to_tsvector('simple', coalesce(module,'')), 'B')
  ||  setweight(to_tsvector('simple', coalesce(message,'')), 'C')
) STORED;
```

调用

```sql
const sql = `
SELECT *
FROM system_log
WHERE log_time BETWEEN $1 AND $2
  AND search_vec @@ websearch_to_tsquery('simple', $3)
ORDER BY log_time DESC
LIMIT $4
`; -- 写成函数

await pg.query(sql, [
  startTime,
  endTime,
  keyword,   // 'error timeout'
  100
]);
```

#### 全文检索 + 中文方案

PG 原生 不懂中文分词，这是事实，但有成熟解法。

##### 1.中文全文检索的三种方案（结论版）

| 方案              | 推荐度 | 说明             |
| ----------------- | ------ | ---------------- |
| simple + 手动切词 | ⭐⭐⭐⭐   | 最稳，工程最常用 |
| zhparser 插件     | ⭐⭐⭐    | 可用，但部署成本 |
| pg_trgm 兜底      | ⭐⭐⭐⭐⭐  | 中文模糊王者     |

##### 2.中文日志的“工程级”做法

核心思想

> 搜索词切在 Node.js
>  PG 只负责索引和匹配

------

Node.js 侧（示例）

```js
function splitChineseKeyword(keyword) {
  return keyword
    .replace(/\s+/g, ' ')
    .split('')
    .join(' ');
}

// “支付失败” → “支 付 失 败”
```

------

PG 查询

```sql
WHERE search_vec @@ plainto_tsquery(
  'simple',
  '支 付 失 败'
)
```

👉 对 日志 / 错误描述 非常够用

------

##### 3.中文 + 权重依然生效

```sql
setweight(to_tsvector('simple', message), 'C')
```

字符被拆，但：

- 出现频率
- 相关度排序

都还能用。

如果你后面真要做：

#### 全文检索 + pg_trgm 组合用法（杀手锏）

这是最重要的一节，也是你区别于普通后端的地方。

------

##### 1️⃣ 为什么要组合？

| 问题     | 全文检索 | pg_trgm |
| -------- | -------- | ------- |
| 中文     | 一般     | 很强    |
| 拼写错误 | ❌        | ✅       |
| 少字     | ❌        | ✅       |
| 多关键词 | ✅        | ❌       |

👉 组合 = 覆盖所有搜索需求

------

##### 2️⃣ 推荐表结构（直接可用）

```sql
CREATE INDEX idx_log_trgm
ON system_log_20251223
USING gin (message gin_trgm_ops);
```

现在你有：

- search_vec → 全文检索
- message gin_trgm_ops → 模糊兜底

------

##### 3️⃣ 推荐查询策略（实战）

✅ 主搜索：全文检索

```sql
WHERE search_vec @@ websearch_to_tsquery('simple', $1)
```

✅ 兜底搜索：trgm

```sql
OR message ILIKE '%' || $1 || '%'
```

------

完整示例

```sql
SELECT *
FROM system_log
WHERE log_time BETWEEN $1 AND $2
  AND (
        search_vec @@ websearch_to_tsquery('simple', $3)
     OR message ILIKE '%' || $3 || '%'
  )
ORDER BY log_time DESC
LIMIT 100;
```

------

##### 4️⃣ 为什么性能还能撑住？

- 分区 → 只扫一天
- search_vec → GIN
- trgm → GIN
- OR 在 同一分区内

👉 这是 PG 少数能承受的 OR 组合场景

#### 表分区

##### 1.为什么日志一定要分区？

结合你现在 & 未来的场景：

- 日志只会 追加
- 查询 99% 带时间条件
- 旧日志几乎不查，但要保留

👉 时间分区 = 天然完美解

##### 2..推荐分区策略

```sql
PARTITION BY RANGE (log_time)
```

为什么不是按月？

- 日志量不稳定
- 单分区过大 → 索引臃肿
- 日级最灵活（删、归档、冷热分离）

##### 3.标准日志主表

```sql
CREATE TABLE system_log (
    id          BIGSERIAL,
    log_time    TIMESTAMPTZ NOT NULL,
    level       TEXT,
    module      TEXT,
    message     TEXT,

    -- 全文检索
    search_vec  tsvector GENERATED ALWAYS AS (
          setweight(to_tsvector('simple', coalesce(level,'')), 'A')
      ||  setweight(to_tsvector('simple', coalesce(module,'')), 'B')
      ||  setweight(to_tsvector('simple', coalesce(message,'')), 'C')
    ) STORED
)
PARTITION BY RANGE (log_time);
```

注意：主表不建索引

##### 4.创建日分区

```sql
CREATE TABLE system_log_20251223
PARTITION OF system_log
FOR VALUES FROM ('2025-12-23') TO ('2025-12-24');
```

##### 5.每个分区必须建立索引

```sql
-- 时间
CREATE INDEX ON system_log_20251223 (log_time);

-- 全文检索
CREATE INDEX ON system_log_20251223
USING gin (search_vec);
```

PG 不会自动给分区建索引

##### 6.查询时会发生什么

```sql
SELECT *
FROM system_log
WHERE log_time BETWEEN '2025-12-23 10:00'
                    AND '2025-12-23 12:00'
  AND search_vec @@ websearch_to_tsquery('simple', 'error timeout');
```

执行计划会是：

- ✅ 只扫描当天分区
- ❌ 不碰其他分区
- 索引全部生效

这叫 Partition Pruning.

##### 7.分区维护

你后面一定会做：

- 定时创建明天分区
- 删除 N 天前分区

```sql
DROP TABLE system_log_20241001;
```

👉 秒删，不走 VACUUM

#### 分区自动化 pg_cron+SQL

- 自动建未来分区
- 自动删过期分区

##### 1-安装插件&修改配置

```sql
--修改配置 需要重启
shared_preload_libraries = 'pg_cron'
cron.database_name = 'your_db_name'
--安装插件
CREATE EXTENSION IF NOT EXISTS pg_cron;
--验证
SELECT * FROM cron.job;
```

##### 2-自动创建"未来N天分区"

命名规范 xxx_log_YYYYMMDD

```sql
system_log_20251224
```

创建分区函数

```sql
CREATE OR REPLACE FUNCTION create_system_log_partition(p_date date)--'2025-12-24'
RETURNS void AS $$
DECLARE
    v_start timestamptz := p_date;
    v_end   timestamptz := p_date + interval '1 day';
    v_table text := format('system_log_%s', to_char(p_date, 'YYYYMMDD'));
BEGIN
--原生 declarative partition
    EXECUTE format(
        'CREATE TABLE IF NOT EXISTS %I
         PARTITION OF system_log
         FOR VALUES FROM (%L) TO (%L)',
        v_table, v_start, v_end
    );

    -- 时间索引
    EXECUTE format(
        'CREATE INDEX IF NOT EXISTS %I ON %I (log_time)',
        v_table || '_log_time_idx', v_table
    );

    -- 全文检索索引
    EXECUTE format(
        'CREATE INDEX IF NOT EXISTS %I
         ON %I USING gin (search_vec)',
        v_table || '_search_idx', v_table
    );
END;
$$ LANGUAGE plpgsql;
--一次性建立未来7天
DO $$
DECLARE
    d date;
BEGIN
    FOR d IN CURRENT_DATE .. CURRENT_DATE + 7 LOOP
        PERFORM create_system_log_partition(d);
    END LOOP;
END $$;
```

用pg_cron创建定时任务自动创建

```sql
SELECT cron.schedule(
    'create_system_log_partition_daily',
    '0 0 * * *',        -- 每天 00:00
    $$SELECT create_system_log_partition(CURRENT_DATE + 7);$$
);
pg_cron
   ↓
每天创建未来第 7 天分区
   ↓
任何时间写日志
   ↓
Postgres 自动路由
```

##### 3-自动删除过期分区

删除函数

```sql
CREATE OR REPLACE FUNCTION drop_old_system_log_partition(p_keep_days int)
RETURNS void AS $$
DECLARE
    v_cutoff date := CURRENT_DATE - p_keep_days;
    r record;
BEGIN
    FOR r IN
        SELECT inhrelid::regclass AS part
        FROM pg_inherits
        WHERE inhparent = 'system_log'::regclass
    LOOP
        IF substring(r.part::text FROM '(\d{8})')::date < v_cutoff THEN
            EXECUTE format('DROP TABLE IF EXISTS %s', r.part);
        END IF;
    END LOOP;
END;
$$ LANGUAGE plpgsql;
```

定时执行

```sql
SELECT cron.schedule(
    'drop_old_system_log_partition',
    '30 0 * * *',   -- 每天 00:30
    $$SELECT drop_old_system_log_partition(90);$$
);
--保留90天的
```

#### 日志冷热分离&归档策略

目标

- 热日志：快、可搜
- 冷日志：便宜、可追溯
- 存储成本 & 性能都可控

##### 1-推荐的日志生命周期

| 阶段 | 时间     | 存储          | 可搜索    |
| ---- | -------- | ------------- | --------- |
| 热   | 0-7天    | pg分区        | √         |
| 温   | 8-30天   | pg分区        | √(慢一点) |
| 冷   | 31-180天 | 文件/对象存储 | ×/离线    |
| 归档 | 180+     | 压缩文件      | ×         |

##### 2-温->冷:导出分区

```sql
COPY system_log_20251101
TO '/data/log_archive/system_log_20251101.csv'
WITH (FORMAT csv, HEADER true);
--工具直接导出
COPY system_log_20251101
TO PROGRAM 'gzip > /data/log_archive/system_log_20251101.csv.gz';

DROP TABLE system_log_20251101;
--导出完删库
```

##### 3-冷日志"临时回查"

```sql
CREATE TEMP TABLE tmp_log_import (
    LIKE system_log INCLUDING ALL
);

COPY tmp_log_import
FROM PROGRAM 'gunzip -c /data/log_archive/system_log_20251101.csv.gz'
WITH (FORMAT csv, HEADER true);
--也就从文件导入
```
