# 230万行客户数据清洗、分层与聚类实战：PySpark Connect 生产级脚本拆解

在客户分析、用户画像和运营分层场景里，`200W~500W` 行数据往往处在一个很尴尬的位置：

- 用 `Pandas` 能做，但很容易吃满内存
- 上完整数仓集群又偏重
- 数据结构还常常不规整，既有 CSV，又夹着 JSON 字符串、混合时间格式、地理坐标文本

这篇文章基于一个真实场景展开：对一份约 **230 万行** 的用户 CSV 做清洗、特征提取、RFM 分层、运营分群和可选的 KMeans 聚类，并通过 **PySpark Connect 4.0.2** 直接连接本地 Spark 服务完成计算。

本文不是概念介绍，而是完整的工程化落地说明：

- 为什么选 `PySpark Connect`
- 这类 CSV 数据怎么做强类型解析
- 混乱时间格式怎么兜底
- 230 万行数据下，哪些参数该保守，哪些地方不要硬上
- 脚本为什么这么组织
- 全量执行时怎么降低把 Spark Connect 服务打挂的概率

## 一、场景与数据特点

这次处理的是一份用户主数据 CSV，规模约 230 万行。字段并不简单，典型特点有四类：

1. 标量字段很多
   例如 `userno`、`gender`、`status`、`credit`、`experience`、`login_times`

2. 多个字段是 JSON 字符串
   例如 `counts_info`、`login_info`、`my_settings`、`push_settings`、`message_settings`

3. 时间格式不统一
   既有：
   - `2025-12-01 13:38:05.115 +00:00`
   - `10/12/2023 22:45:36.173+00`
   - `2025-04-30T07:28:29.732981+08:00`

4. 有文本型地理坐标
   例如：

```text
POINT (119.9740719795227 26.68752908706665)
```

这意味着如果直接 `inferSchema + Pandas`，很快就会遇到三个问题：

- 类型推断不稳定
- 时间解析失败率高
- JSON 字段后续分析不方便

所以这类任务更适合直接走 Spark 的强类型清洗链路。

## 二、为什么选 PySpark Connect

这里的运行环境是：

- `conda` 环境：`spark_work`
- `PySpark` 版本：`4.0.2`
- `Spark Connect Server` 版本：`4.0.2`
- Connect 地址：`sc://localhost:15002`

相比传统的本地 `SparkSession`，`Spark Connect` 的优点主要有三个：

1. Python 侧更轻
   Python 进程不直接嵌 JVM，开发体验更干净。

2. 计算与客户端解耦
   Python 脚本只负责生成逻辑计划，真正执行交给 Spark 服务端。

3. 更适合本地开发 + 独立服务端
   尤其适合“本机有 Spark 服务，Python 代码独立维护”的工作方式。

但它也有两个必须接受的现实：

1. 路径是服务端视角
   所以读写路径建议一律使用绝对路径。

2. 服务端更脆弱
   如果脚本一上来就做大表 `show()`、多轮 `collect()`、`persist()`，更容易把服务端打到不稳定。

这也是后面脚本里默认增加：

- `--skip-preview`
- `--skip-console-report`

这两个参数的原因。

## 三、整体方案设计

脚本分成五个阶段：

1. `parse_args`
   统一管理输入路径、输出路径、分区、聚类参数。

2. `build_spark`
   用 `SparkSession.builder.remote(...)` 连接 Spark Connect 服务，并设置时区、AQE、Arrow 等参数。

3. `add_base_features`
   做基础类型清洗、JSON 解析、时间解析、地域拆分、经纬度提取和行为特征构建。

4. `enrich_segments`
   基于清洗后的宽表构建：
   - 活跃状态分层
   - 价值分层
   - 参与度分层
   - 生命周期分层
   - RFM 分层

5. `save_reports`
   输出：
   - 明细特征表
   - 汇总指标
   - 各类分层分布
   - 地域/设备/系统/版本/月份趋势
   - 可选聚类结果

## 四、脚本里的关键设计点

### 1. 强类型 Schema 解析 JSON 字段

CSV 中的多个字段其实是 JSON 字符串。最稳的做法不是保留原字符串，而是先定义 Schema，再 `from_json`。

例如 `counts_info`：

```python
COUNTS_INFO_SCHEMA = T.StructType(
    [
        T.StructField("postCount", T.IntegerType(), True),
        T.StructField("catchRecord", T.IntegerType(), True),
        T.StructField("activityCount", T.IntegerType(), True),
        T.StructField("favoriteCount", T.IntegerType(), True),
    ]
)
```

然后解析：

```python
.withColumn("counts_info_obj", F.from_json("counts_info", COUNTS_INFO_SCHEMA))
.withColumn("post_count", F.coalesce(F.col("counts_info_obj.postCount"), F.lit(0)))
```

这样做有两个直接收益：

- 后续特征工程能直接按列操作
- 空值和异常值的处理更统一

### 2. 混合时间格式要用兜底链路

这类业务 CSV 最常见的坑不是空值，而是时间格式混乱。

脚本中的 `parse_multi_format_timestamp()` 用 `F.coalesce()` 依次尝试多种格式，覆盖了：

- `yyyy-MM-dd HH:mm:ss.SSS +00:00`
- `yyyy-MM-dd HH:mm:ss.SSS +00`
- `d/M/yyyy HH:mm:ss.SSS+00`
- `yyyy-MM-dd'T'HH:mm:ss.SSSSSS+08:00`

核心思路不是“找到唯一格式”，而是“做一个格式兜底网”：

```python
return F.coalesce(
    F.to_timestamp(F.col(col_name), "yyyy-MM-dd HH:mm:ss.SSS XXX"),
    F.to_timestamp(F.col(col_name), "yyyy-MM-dd HH:mm:ss.SSS XX"),
    F.to_timestamp(F.col(col_name), "yyyy-MM-dd HH:mm:ss.SSS X"),
    ...
)
```

这是整条链路能跑通的关键之一。前一版样本运行时，大量 `register_date` 为 `NULL`，就是因为没覆盖 `+00` 这类时区格式，后面补齐模板后才恢复正常。

### 3. 主活跃时间不能只看一个字段

用户活跃并不一定都体现在同一个字段里。

脚本用了一个保守但实用的策略：

```python
.withColumn(
    "primary_last_active_ts",
    F.coalesce("last_use_ts", "last_login_ts", "login_info_last_ts", "edit_time_ts", "createtime_ts"),
)
```

优先级的含义是：

- 优先取最近使用时间
- 没有就取最近登录时间
- 再退到嵌套登录信息
- 再退到编辑时间
- 最后退到创建时间

这样 `recency_days` 才不会因为单字段缺失而大面积失真。

### 4. 画像特征尽量用内置算子，不用 Python UDF

像下面这些特征都完全可以用 Spark 内置表达式完成：

- 地域拆分
- 经纬度抽取
- 资料完整度
- 活跃度评分

例如活跃度：

```python
.withColumn(
    "activity_score",
    F.col("post_count") * 5
    + F.col("activity_count") * 4
    + F.col("catch_record_count") * 3
    + F.col("favorite_count") * 2
    + F.coalesce(F.col("login_times"), F.lit(0)),
)
```

这类写法的好处是：

- Catalyst 能优化
- 执行全在 JVM 端完成
- 避免 Python UDF 的序列化开销

## 五、分层模型设计

这份脚本并没有直接上复杂模型，而是先用业务上最稳定的规则分层。

### 1. 活跃状态分层

按 `recency_days` 分成：

- `active_7d`
- `active_30d`
- `sleeping_90d`
- `churn_risk`

适合做：

- 近 7 日活跃用户运营
- 沉睡用户召回
- 流失风险监控

### 2. 价值分层

按 `is_vip`、`credit`、`experience` 分为：

- `high_value`
- `mid_value`
- `base_value`

这是典型的运营分层，不要求绝对科学，但要求口径稳定、容易解释。

### 3. 参与度分层

按 `activity_score` 分成：

- `high_engagement`
- `mid_engagement`
- `low_engagement`

### 4. 生命周期分层

同时结合 `tenure_days` 和 `recency_days`：

- `new_active`
- `retained`
- `sleeping`
- `lost`

### 5. RFM 分层

脚本里实现的是轻量版 RFM：

- `R`: `recency_days`
- `F`: `login_times`
- `M`: `is_vip / credit / experience`

最后拼成：

```python
rfm_segment = R + "_" + F + "_" + M
```

比如：

```text
R1_7d_F4_one_time_M3_base
R4_90d_plus_F3_low_M3_base
```

这种标签很适合直接丢给运营或 BI 看板。

## 六、为什么聚类放在“可选项”

脚本支持 KMeans，但没有默认强制开启。这是有意为之。

原因很简单：

1. 规则分层是业务稳定层
   运营和产品通常更先需要这个。

2. 聚类是探索层
   对数据质量、特征质量、服务端资源更敏感。

3. Spark Connect 服务端稳定性要优先考虑
   在服务端不够稳时，先把规则报表跑通，再开聚类更合理。

当前聚类输入特征包括：

- `credit`
- `experience`
- `login_times`
- `post_count`
- `catch_record_count`
- `activity_count`
- `favorite_count`
- `record_score`
- `serial_sign_in_count`
- `total_sign_in_count`
- `activity_score`
- `profile_completeness_score`
- `recency_days`
- `tenure_days`
- `push_receive_all`

然后：

1. 缺失值补 `0.0`
2. `VectorAssembler`
3. `StandardScaler`
4. `KMeans`

如果你的 Spark Connect 服务比较稳，可以把：

```bash
--cluster-sample-fraction 0.2
```

逐步调高。不要一开始就默认全量聚类，更不要在服务端不稳时硬顶 `1.0`。

## 七、230万行数据下的参数建议

这一块不应该给“秒级”这种绝对承诺，性能取决于：

- Spark 服务端资源
- 本地磁盘吞吐
- CSV 本身的列宽
- 是否开启聚类
- 是否同时做大量 `coalesce(1)` 输出

更稳妥的建议是：

### 1. 默认先跑规则分层，不跑聚类

```bash
--skip-preview --skip-console-report
```

先把明细表和报表导出来，确认服务端稳定。

### 2. `shuffle-partitions` 和 `repartition` 不要拍脑袋

当前脚本默认：

- `--shuffle-partitions 400`
- `--repartition 600`

这不是“通用最优值”，只是相对稳妥的默认值。你可以这样理解：

- 数据列多、转换多、服务端资源一般：先保守大一点
- 服务端 CPU 和内存充足：可以逐步压小
- 本地机单机且任务不重：常见可尝试 `100~300`

不要把“230 万行”机械等同于“分区一定很小”。Spark Connect 稳定性优先。

### 3. 明细表和报表要区别处理

脚本里：

- 明细宽表输出为 `parquet`
- 报表类结果输出为单文件 `csv`

这是因为：

- 明细表适合列式存储
- 汇总报表体量小，单文件更方便查看

但不要对超大明细表做 `coalesce(1)`。

## 八、实战里踩过的坑

这部分比“理论最优”更重要。

### 1. SparkContext stopped

全量执行时遇到过：

```text
Cannot call methods on a stopped SparkContext
```

这不是 Python 语法问题，而是 Spark Connect 服务端在执行过程中已经不可用了。可能原因：

- 服务端被重启
- 内存/资源不足
- 脚本启动时做了过多控制台输出
- 大表 `persist`、`show`、多轮 action 叠加

因此脚本后面做了两件事：

- 去掉默认 `persist`
- 增加 `--skip-preview` 和 `--skip-console-report`

### 2. 时间解析漏格式

第一次样本运行时，大量 `register_date` 为空。根因是没有覆盖 `+00` 这类偏短时区格式。

所以时间解析必须做成“模板链”，不能只写一两种格式。

### 3. Spark Connect 下路径一定要绝对路径

相对路径容易踩服务端工作目录不一致的问题。建议统一使用：

```bash
--input /abs/path/to/file.csv
--output /abs/path/to/output_dir
```

def safe_int(col_name: str) -> F.Column:
    """安全地将字符串列转换为整型，处理掉空格和空字符串，防止转出脏数据"""
    return F.when(F.trim(F.col(col_name)) == "", None).otherwise(F.col(col_name).cast("int"))


def parse_multi_format_timestamp(col_name: str) -> F.Column:
    """
    【极端复杂场景】使用 coalesce 盲猜并兼容几十种不同的时间字符串格式。
    从上到下按优先级匹配，一旦前面的格式解析成功，后面的就会被跳过。
    """
    return F.coalesce(
        # 格式群 1：带标准时区尾缀 (如 XXX, XX, X) 的 yyyy-MM-dd 型
        F.to_timestamp(F.col(col_name), "yyyy-MM-dd HH:mm:ss.SSS XXX"),
        F.to_timestamp(F.col(col_name), "yyyy-MM-dd HH:mm:ss.SSS XX"),
        F.to_timestamp(F.col(col_name), "yyyy-MM-dd HH:mm:ss.SSS X"),
        F.to_timestamp(F.col(col_name), "yyyy-MM-dd HH:mm:ss.SS XXX"),
        F.to_timestamp(F.col(col_name), "yyyy-MM-dd HH:mm:ss.SS XX"),
        F.to_timestamp(F.col(col_name), "yyyy-MM-dd HH:mm:ss.SS X"),
        F.to_timestamp(F.col(col_name), "yyyy-MM-dd HH:mm:ss.S XXX"),
        F.to_timestamp(F.col(col_name), "yyyy-MM-dd HH:mm:ss.S XX"),
        F.to_timestamp(F.col(col_name), "yyyy-MM-dd HH:mm:ss.S X"),
        # 格式群 2：英式/日式斜杠非补零型日期 (如 d/M/yyyy)
        F.to_timestamp(F.col(col_name), "d/M/yyyy HH:mm:ss.SSSXXX"),
        F.to_timestamp(F.col(col_name), "d/M/yyyy HH:mm:ss.SSSXX"),
        F.to_timestamp(F.col(col_name), "d/M/yyyy HH:mm:ss.SSSX"),
        F.to_timestamp(F.col(col_name), "d/M/yyyy H:mm:ss.SSSXXX"),
        F.to_timestamp(F.col(col_name), "d/M/yyyy H:mm:ss.SSSXX"),
        F.to_timestamp(F.col(col_name), "d/M/yyyy H:mm:ss.SSSX"),
        F.to_timestamp(F.col(col_name), "d/M/yyyy HH:mm:ss.SSXXX"),
        F.to_timestamp(F.col(col_name), "d/M/yyyy HH:mm:ss.SSXX"),

        F.to_timestamp(F.col(col_name), "d/M/yyyy H:mm:ss.SSX"),
        F.to_timestamp(F.col(col_name), "d/M/yyyy HH:mm:ss.SXXX"),
        F.to_timestamp(F.col(col_name), "d/M/yyyy HH:mm:ss.SXX"),
        F.to_timestamp(F.col(col_name), "d/M/yyyy HH:mm:ss.SX"),
        F.to_timestamp(F.col(col_name), "d/M/yyyy H:mm:ss.SXXX"),
        F.to_timestamp(F.col(col_name), "d/M/yyyy H:mm:ss.SXX"),
        F.to_timestamp(F.col(col_name), "d/M/yyyy H:mm:ss.SX"),
        # 格式群 3：带 'T' 分隔符的 ISO 标准时间戳格式 (常见于前端或各端埋点上报)
        F.to_timestamp(F.col(col_name), "yyyy-MM-dd'T'HH:mm:ss.SSSSSSXXX"),
        F.to_timestamp(F.col(col_name), "yyyy-MM-dd'T'HH:mm:ss.SSSSSXXX"),

def load_raw_csv(spark: SparkSession, path: str) -> DataFrame:
    """加载原始 CSV 数据，配置防数据串行的各种引号逃逸参数"""
    return (

        .option("multiLine", False)  # 严格单行模式提高读取性能

        .option("mode", "PERMISSIVE")  # 遇到脏行时将字段置为空，而不是直接让整个任务崩溃
        .csv(path)
    )

        .withColumn("login_info_obj", F.from_json("login_info", LOGIN_INFO_SCHEMA))

        # --- 5.4 从 Struct 结构体中抽取具体的具体指标 ---
        .withColumn(

                F.to_timestamp(F.col("login_info_obj.lastTime"), "yyyy-MM-dd'T'HH:mm:ssX"),
            ),

        .withColumn("post_count", F.coalesce(F.col("counts_info_obj.postCount"), F.lit(0)))
        .withColumn("catch_record_count", F.coalesce(F.col("counts_info_obj.catchRecord"), F.lit(0)))
        .withColumn("activity_count", F.coalesce(F.col("counts_info_obj.activityCount"), F.lit(0)))
        .withColumn("favorite_count", F.coalesce(F.col("counts_info_obj.favoriteCount"), F.lit(0)))
        .withColumn("privacy_friend_check", F.coalesce(F.col("my_settings_obj.privacy.friendCheck"), F.lit(0)))
        .withColumn("privacy_mobile_search", F.coalesce(F.col("my_settings_obj.privacy.mobileSearch"), F.lit(0)))
        .withColumn("privacy_userno_search", F.coalesce(F.col("my_settings_obj.privacy.usernoSearch"), F.lit(0)))
        .withColumn("privacy_commend_contacts", F.coalesce(F.col("my_settings_obj.privacy.commendContacts"), F.lit(0)))
        # 计算推送通知开启的总项数
        .withColumn(
            "push_receive_all",
            F.coalesce(F.col("push_settings_obj.group"), F.lit(0))
            + F.coalesce(F.col("push_settings_obj.action"), F.lit(0))
            + F.coalesce(F.col("push_settings_obj.friend"), F.lit(0))
            + F.coalesce(F.col("push_settings_obj.review"), F.lit(0))
            + F.coalesce(F.col("push_settings_obj.subscribe"), F.lit(0)),

        .withColumn("message_receive", F.coalesce(F.col("message_settings_obj.receive"), F.lit(0)))
        .withColumn("message_display", F.coalesce(F.col("message_settings_obj.display"), F.lit(0)))
        .withColumn("message_warning", F.coalesce(F.col("message_settings_obj.warning"), F.lit(0)))
        # --- 5.5 业务特征衍生衍生 ---
        # 统一多端混杂的活跃时间，推导出一个最能代表用户最后一次现身的“主活跃时间戳”
        .withColumn(
            "primary_last_active_ts",
            F.coalesce("last_use_ts", "last_login_ts", "login_info_last_ts", "edit_time_ts", "createtime_ts"),
        )
        .withColumn("register_date", F.to_date("createtime_ts"))
        .withColumn("last_active_date", F.to_date("primary_last_active_ts"))
        # 解析地理围栏编码 (例如把 "cn.gd.gz" 拆成国家、省份、城市)
        .withColumn("country_code", F.element_at(F.split(F.col("hasc"), r"\."), 1))
        .withColumn("province_code", F.element_at(F.split(F.col("hasc"), r"\."), 2))
        .withColumn("city_code", F.element_at(F.split(F.col("hasc"), r"\."), 3))
        # 正则提取 GIS 空间字段 POINT (经度 纬度)
        .withColumn(
            "longitude",
            F.regexp_extract("poslocation", r"POINT\s*\(([-0-9.]+)\s+[-0-9.]+\)", 1).cast("double"),
        )
        .withColumn(
            "latitude",
            F.regexp_extract("poslocation", r"POINT\s*\([-0-9.]+\s+([-0-9.]+)\)", 1).cast("double"),
        )
        # 统计资料完整度 (有值计1分，无值计0分)
        .withColumn("has_avatar", F.when((F.col("avatar").isNotNull()) & (F.trim("avatar") != ""), 1).otherwise(0))
        .withColumn("has_sign", F.when((F.col("sign").isNotNull()) & (F.trim("sign") != ""), 1).otherwise(0))
        .withColumn("has_email", F.when((F.col("email").isNotNull()) & (F.trim("email") != ""), 1).otherwise(0))
        .withColumn("has_address", F.when((F.col("address").isNotNull()) & (F.trim("address") != ""), 1).otherwise(0))
        .withColumn("has_website", F.when((F.col("website").isNotNull()) & (F.trim("website") != ""), 1).otherwise(0))
        .withColumn("profile_completeness_score",
                    F.expr("has_avatar + has_sign + has_email + has_address + has_website"))
        # 综合活跃度加权评分 (发帖权重最高，登录权重最低)
        .withColumn(
            "activity_score",
            F.col("post_count") * 5
            + F.col("activity_count") * 4
            + F.col("catch_record_count") * 3
            + F.col("favorite_count") * 2
            + F.coalesce(F.col("login_times"), F.lit(0)),
        )
    )


# ==============================================================================
# 6. 用户运营规则分群 (Rule-based Segmentation)
# ==============================================================================
def enrich_segments(df: DataFrame) -> DataFrame:
    """基于固定的运营业务规则，对用户进行多维度的标签化分群"""

    # 【⚠️警告】这里执行了 .collect() 触发了 Action 操作。
    # 它会将全局最大活跃日期和总人数拉回到 Python 驱动端，用于后续作为固定的常量进行广播计算。
    stats_row = df.agg(
        F.max(F.to_date("primary_last_active_ts")).alias("analysis_date"),
        F.count("*").alias("total_users"),
    ).collect()[0]

    analysis_date = stats_row["analysis_date"]
    total_users = stats_row["total_users"]

    if analysis_date is None:
        raise ValueError("输入数据中未找到有效的活跃时间戳。")

    return (
        df.withColumn("analysis_date", F.lit(analysis_date).cast("date"))
        .withColumn("dataset_total_users", F.lit(total_users))
        # 计算核心指标：静默天数 (Recency) 和 在册天数 (Tenure)
        .withColumn("recency_days", F.datediff(F.col("analysis_date"), F.col("last_active_date")))
        .withColumn("tenure_days", F.datediff(F.col("analysis_date"), F.col("register_date")))
        # 6.1 R (Recency) 静默时间段分群
        .withColumn(
            "recency_segment",
            F.when(F.col("recency_days").isNull(), "unknown")
            .when(F.col("recency_days") <= 7, "R1_7d")
            .when(F.col("recency_days") <= 30, "R2_30d")
            .when(F.col("recency_days") <= 90, "R3_90d")
            .otherwise("R4_90d_plus"),
        )
        # 6.2 F (Frequency) 频次段分群
        .withColumn(
            "frequency_segment",
            F.when(F.col("login_times") >= 50, "F1_high")
            .when(F.col("login_times") >= 10, "F2_mid")
            .when(F.col("login_times") >= 2, "F3_low")
            .otherwise("F4_one_time"),
        )
        # 6.3 M (Monetary) 价值段分群 (这里用高资产/高VIP来替代金额)
        .withColumn(
            "monetary_segment",
            F.when((F.col("is_vip") == 1) | (F.col("credit") >= 1000) | (F.col("experience") >= 500), "M1_high")
            .when((F.col("credit") >= 300) | (F.col("experience") >= 100), "M2_mid")
            .otherwise("M3_base"),
        )
        # 6.4 用户活跃状态精细化分群
        .withColumn(
            "active_status_segment",
            F.when(F.col("recency_days").isNull(), "unknown")
            .when(F.col("recency_days") <= 7, "active_7d")
            .when(F.col("recency_days") <= 30, "active_30d")
            .when(F.col("recency_days") <= 90, "sleeping_90d")
            .otherwise("churn_risk"),
        )
        # 6.5 高/中/低价值客群
        .withColumn(
            "value_segment",
            F.when((F.col("is_vip") == 1) | (F.col("credit") >= 1000) | (F.col("experience") >= 500), "high_value")
            .when((F.col("credit") >= 300) | (F.col("experience") >= 100), "mid_value")
            .otherwise("base_value"),
        )
        # 6.6 互动深度分群
        .withColumn(
            "engagement_segment",
            F.when(F.col("activity_score") >= 50, "high_engagement")
            .when(F.col("activity_score") >= 10, "mid_engagement")
            .otherwise("low_engagement"),
        )
        # 6.7 经典的生命周期阶段模型划分
        .withColumn(
            "lifecycle_segment",
            F.when((F.col("tenure_days") <= 30) & (F.col("recency_days") <= 7), "new_active")
            .when((F.col("tenure_days") > 30) & (F.col("recency_days") <= 30), "retained")
            .when((F.col("recency_days") > 30) & (F.col("recency_days") <= 90), "sleeping")
            .when(F.col("recency_days") > 90, "lost")
            .otherwise("unknown"),
        )
        # 6.8 交叉矩阵字段拼接 (用于多维下钻切片分析)
        .withColumn(
            "customer_segment",
            F.concat_ws("_", "value_segment", "engagement_segment", "active_status_segment"),
        )
        .withColumn(
            "rfm_segment",
            F.concat_ws("_", "recency_segment", "frequency_segment", "monetary_segment"),
        )
    )


# ==============================================================================
# 7. 数据持久化封装 (I/O)
# ==============================================================================
def write_df(df: DataFrame, path: str, fmt: str = "parquet", single_file: bool = False) -> None:
    """统一的数据写盘落地工具函数"""
    # 【⚠️注意】如果开了 single_file=True，数据会被强行规整到一个分区。
    # 绝对不能对百万级别的未聚合宽表调用此参数，否则单节点会直接 OOM 崩溃。
    out_df = df.coalesce(1) if single_file else df
    writer = out_df.write.mode("overwrite")
    if fmt == "csv":
        writer = writer.option("header", True)
    writer.format(fmt).save(path)


# ==============================================================================
# 8. 统计分析与报表矩阵生成矩阵生成
# ==============================================================================
def summary_metrics(df: DataFrame) -> DataFrame:
    """计算全局大盘核心摘要指标（均值、中位数、去重人数等）"""
    return df.agg(
        F.max("dataset_total_users").alias("total_users"),
        F.countDistinct("userno").alias("distinct_users"),
        F.sum(F.when(F.col("status") == 1, 1).otherwise(0)).alias("active_status_users"),
        F.sum("is_vip").alias("vip_users"),
        F.avg("credit").alias("avg_credit"),
        F.expr("percentile_approx(credit, 0.5)").alias("median_credit"),  # 百分位数近似，代表中位数
        F.avg("experience").alias("avg_experience"),
        F.avg("login_times").alias("avg_login_times"),
        F.avg("activity_score").alias("avg_activity_score"),
        F.avg("recency_days").alias("avg_recency_days"),
        F.expr("percentile_approx(recency_days, 0.5)").alias("median_recency_days"),
        F.avg("tenure_days").alias("avg_tenure_days"),
        F.expr("percentile_approx(tenure_days, 0.5)").alias("median_tenure_days"),
        F.avg("profile_completeness_score").alias("avg_profile_completeness_score"),
    )


def segment_distribution(df: DataFrame, segment_col: str) -> DataFrame:
    """通用工具：统计任意标签群的用户覆盖量及全局占比"""
    return (
        df.groupBy(segment_col)
        .agg(F.count("*").alias("user_count"), F.max("dataset_total_users").alias("total_users"))
        .withColumn("ratio", F.round(F.col("user_count") / F.col("total_users"), 6))
        .drop("total_users")
        .orderBy(F.desc("user_count"))
    )


def build_dimension_reports(df: DataFrame) -> dict[str, DataFrame]:
    """批量对常用的用户基础属性进行维度聚合 (Group By)"""
    return {
        "province_distribution": df.groupBy("province_code").count().orderBy(F.desc("count")),
        "city_distribution": df.groupBy("city_code").count().orderBy(F.desc("count")),
        "gender_distribution": df.groupBy("gender").count().orderBy("gender"),
        "device_distribution": df.groupBy("device").count().orderBy(F.desc("count")),
        "system_distribution": df.groupBy("system").count().orderBy(F.desc("count")),
        "domain_distribution": df.groupBy("domain").count().orderBy(F.desc("count")),
        "language_distribution": df.groupBy("language").count().orderBy(F.desc("count")),
        "usertype_distribution": df.groupBy("usertype").count().orderBy(F.desc("count")),
        "app_version_distribution": df.groupBy("app_version").count().orderBy(F.desc("count")),
        # 按年月统计注册趋势和最后活跃趋势
        "register_month_trend": df.groupBy(
            F.date_format("register_date", "yyyy-MM").alias("register_month")).count().orderBy("register_month"),
        "active_month_trend": df.groupBy(
            F.date_format("last_active_date", "yyyy-MM").alias("active_month")).count().orderBy("active_month"),
    }


# ==============================================================================
# 9. ML 模块: KMeans 聚类用户画像
# ==============================================================================
def cluster_users(df: DataFrame, k: int, sample_fraction: float) -> tuple[DataFrame, DataFrame]:
    """使用 MLlib 机器学习管道对用户特征实施标准化并执行 KMeans 聚类分析"""

    # 选定参与无监督聚类的 15 个核心连续数值型特征
    numeric_cols = [
        "credit", "experience", "login_times", "post_count",
        "catch_record_count", "activity_count", "favorite_count",
        "record_score", "serial_sign_in_count", "total_sign_in_count",
        "activity_score", "profile_completeness_score", "recency_days",
        "tenure_days", "push_receive_all"
    ]

    # 数值列安全容错处理，将 Null 填充为 0.0，避免 ML 算子罢工
    feature_df = df
    for col_name in numeric_cols:
        feature_df = feature_df.withColumn(col_name, F.coalesce(F.col(col_name).cast("double"), F.lit(0.0)))

    # 对全量数据采样。百万级单机运行建议传 0.1~0.2 保护 Mac 内存
    feature_df = feature_df.sample(withReplacement=False, fraction=sample_fraction, seed=42)

    # 将多列数值拼装成单向量列 (Vector)
    assembler = VectorAssembler(inputCols=numeric_cols, outputCol="features_raw")
    transformed = assembler.transform(feature_df)

    # 数据标准化 (Z-Score: 均值为0, 方差为1)，消除不同特征因数量级相差悬殊带来的权重失真
    scaler = StandardScaler(inputCol="features_raw", outputCol="features", withStd=True, withMean=True)
    scaled = scaler.fit(transformed).transform(transformed)

    # 训练 KMeans 模型并预测
    model = KMeans(k=k, seed=42, featuresCol="features", predictionCol="cluster_id").fit(scaled)
    predicted = model.transform(scaled).select("userno", "cluster_id")

    # 反查聚合：将聚类标签 ID 重新 Join 回原数据集，计算每个类别的均值画像指标
    cluster_profile = (
        predicted.join(df, on="userno", how="inner")
        .groupBy("cluster_id")
        .agg(
            F.count("*").alias("user_count"),
            F.avg("credit").alias("avg_credit"),
            F.avg("experience").alias("avg_experience"),
            F.avg("login_times").alias("avg_login_times"),
            F.avg("activity_score").alias("avg_activity_score"),
            F.avg("recency_days").alias("avg_recency_days"),
            F.avg("tenure_days").alias("avg_tenure_days"),
            F.avg("is_vip").alias("vip_ratio"),
        )
        .orderBy("cluster_id")
    )
    return predicted, cluster_profile


# ==============================================================================
# 10. 管道落盘控制器
# ==============================================================================
def save_reports(df: DataFrame, output_dir: str, run_cluster: bool, cluster_k: int,
                 cluster_sample_fraction: float) -> None:
    """集中调度并持久化整个报表矩阵文件的函数"""

    # 限制导出的特征明细宽表的列，剔除中间使用的 Object 临时列，只保留干净的干净字段
    detail_cols = [
        "userno", "nickname", "mobile", "gender", "status", "language",
        "usertype", "domain", "device", "system", "app_version",
        "country_code", "province_code", "city_code", "longitude", "latitude",
        "register_date", "last_active_date", "recency_days", "tenure_days",
        "login_times", "credit", "experience", "is_vip", "post_count",
        "catch_record_count", "activity_count", "favorite_count", "activity_score",
        "profile_completeness_score", "active_status_segment", "value_segment",
        "engagement_segment", "lifecycle_segment", "customer_segment", "rfm_segment"
    ]

    # 10.1 保存全量明细大表 (Parquet分布式存储)
    write_df(df.select(*detail_cols), f"{output_dir}/detail_user_features", fmt="parquet")

    # 10.2 保存大盘及分群分布报告 (CSV单文件)
    write_df(summary_metrics(df), f"{output_dir}/summary_metrics", fmt="csv", single_file=True)
    write_df(segment_distribution(df, "active_status_segment"), f"{output_dir}/segment_active_status", fmt="csv",
             single_file=True)
    write_df(segment_distribution(df, "value_segment"), f"{output_dir}/segment_value", fmt="csv", single_file=True)
    write_df(segment_distribution(df, "engagement_segment"), f"{output_dir}/segment_engagement", fmt="csv",
             single_file=True)
    write_df(segment_distribution(df, "lifecycle_segment"), f"{output_dir}/segment_lifecycle", fmt="csv",
             single_file=True)
    write_df(segment_distribution(df, "customer_segment"), f"{output_dir}/segment_customer", fmt="csv",
             single_file=True)
    write_df(segment_distribution(df, "rfm_segment"), f"{output_dir}/segment_rfm", fmt="csv", single_file=True)

    # 10.3 循环导出多维度维度矩阵报告 (CSV单文件)
    for report_name, report_df in build_dimension_reports(df).items():
        write_df(report_df, f"{output_dir}/{report_name}", fmt="csv", single_file=True)

    # 10.4 导出交叉下钻矩阵分析
    province_segment = (
        df.groupBy("province_code", "customer_segment")
        .agg(F.count("*").alias("user_count"))
        .orderBy(F.desc("user_count"))
    )
    write_df(province_segment, f"{output_dir}/province_customer_segment", fmt="csv", single_file=True)

    # 10.5 导出核心核心 VIP 画像对比
    vip_profile = (
        df.groupBy("is_vip")
        .agg(
            F.count("*").alias("user_count"),
            F.avg("credit").alias("avg_credit"),
            F.avg("experience").alias("avg_experience"),
            F.avg("activity_score").alias("avg_activity_score"),
            F.avg("recency_days").alias("avg_recency_days"),
            F.avg("tenure_days").alias("avg_tenure_days"),
        )
        .orderBy("is_vip")
    )
    write_df(vip_profile, f"{output_dir}/vip_profile", fmt="csv", single_file=True)

    # 10.6 条件触发 KMeans 算法运行
    if run_cluster:
        cluster_result, cluster_profile = cluster_users(df, cluster_k, cluster_sample_fraction)
        write_df(cluster_result, f"{output_dir}/cluster_result", fmt="parquet")
        write_df(cluster_profile, f"{output_dir}/cluster_profile", fmt="csv", single_file=True)


# ==============================================================================
# 11. 主程序入口程序入口
# ==============================================================================
def main() -> None:
    args = parse_args()
    spark = build_spark(args)

    # 加载流并组装转换链逻辑 (此时底层仅仅是生成 DAG 逻辑图，尚未引发实质计算)
    raw_df = load_raw_csv(spark, args.input)
    customer_df = enrich_segments(add_base_features(raw_df, args.repartition))

    print("===== Spark Connect Version =====")
    print(spark.version)

    # 根据传参决定是否触发 show() 进行预览 (show 属于 Action 算子)
    if not args.skip_preview:
        print("===== Data Preview =====")
        customer_df.select(
            "userno", "province_code", "city_code", "register_date",
            "last_active_date", "recency_days", "tenure_days",
            "activity_score", "customer_segment", "rfm_segment"
        ).show(args.preview_rows, truncate=False)

    if not args.skip_console_report:
        print("===== Summary Metrics =====")
        summary_metrics(customer_df).show(truncate=False)

        print("===== Active Segment Distribution =====")
        segment_distribution(customer_df, "active_status_segment").show(truncate=False)

        print("===== Value Segment Distribution =====")
        segment_distribution(customer_df, "value_segment").show(truncate=False)

    # 全量下发任务并执行持久化落盘
    save_reports(
        customer_df,
        args.output,
        args.run_cluster,
        args.cluster_k,
        args.cluster_sample_fraction,
    )

    # 安全停用并注销进程，还内存给系统
    spark.stop()


if __name__ == "__main__":
    main()
