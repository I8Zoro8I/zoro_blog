分布式库 ​

Citus 是 PostgreSQL 的一个开源扩展，它通过分片（Sharding）和复制（Replication）将普通的 PostgreSQL 变成一个分布式数据库系统。

Citus 架构 ​

md

                │  (路由节点)  │

        │              │              │

│   Shards     │ │   Shards    │ │   Shards    │

三、逻辑步骤完全一样
CREATE EXTENSION citus;
SELECT citus_add_node('worker_ip', 5432);
SELECT create_distributed_table(...);


三、初始化 Citus（最关键一步）
1️⃣ 进入 Coordinator
docker exec -it citus_coordinator psql -U postgres -d demo

2️⃣ 启用 citus 扩展
CREATE EXTENSION citus;

3️⃣ 添加 worker 节点
SELECT citus_add_node('worker1', 5432);
SELECT citus_add_node('worker2', 5432);


验证：

SELECT * FROM citus_get_active_worker_nodes();


看到 2 行 = 成功 ✅

四、创建分布式表（真正开始“分担压力”）
1️⃣ 建普通表
CREATE TABLE system_log (
    id BIGSERIAL,
    tenant_id INT,
    message TEXT,
    created_at TIMESTAMPTZ DEFAULT now()
);

2️⃣ 变成分布式表
SELECT create_distributed_table(
    'system_log',
    'tenant_id'
);


🎯 这一刻：

表数据开始分散

角色
Coordinator 接 SQL、解析、路由、合并结果
Worker 真正存数据 & 执行 SQL

关键点：

  • Coordinator 一般不存数据
  • 所有 shard 表都是普通 PostgreSQL 表
  • Citus 是 Postgres Extension，不是独立数据库

一个表在 Citus 里长什么样 ​

逻辑上你看到的是：

sql
system_log
1

物理上实际是：

text
system_log_102008
system_log_102009
system_log_102010
...
1
2
3
4

这些 shard：

  • 分布在不同 worker
  • Citus 自动管理
  • 你不直接操作

分布式表 vs 参考表 ​

分布式表（Distributed Table） ​

数据会被“拆分”到多个 worker

创建方式

sql
SELECT create_distributed_table(
    'system_log',
    'tenant_id'  -- 分片键
);
1
2
3
4

特点

点说明
数据按分片键 hash
写入路由到单个 shard
扩展水平扩展
JOIN⚠️ 有限制

典型适合

  • 日志表
  • 订单表
  • 观察记录
  • 用户行为流水
参考表（Reference Table） ​

整表复制到所有 worker

创建方式

sql
SELECT create_reference_table('user');
1

特点

点说明
数据每个 worker 一份
JOIN本地 JOIN（无跨节点）
写入广播写入
数据量必须小

典型适合

  • 用户表（基础信息）
  • 配置表
  • 字典表
  • 枚举 / 状态表

📌 一句话记忆法

会无限长的表 → 分布式表
小而常 JOIN 的表 → 参考表

分布式 JOIN 的限制 ​

1-Citus 最理想的 JOIN（完全下推） ​

条件

  • 都是 分布式表
  • 分片键相同
  • JOIN 条件 = 分片键
sql
orders (tenant_id)
order_items (tenant_id)

SELECT *
FROM orders o
JOIN order_items i
  ON o.tenant_id = i.tenant_id
WHERE o.tenant_id = 123;
1
2
3
4

7
8

✅ JOIN 在 worker 本地完成
🚀 性能最好

2-分布式表+参考表 ​
sql
logs (app_id)      -- 分布式
apps (id)          -- 参考表

SELECT *
FROM logs l
JOIN apps a ON l.app_id = a.id;
1
2
3
4
5
6

每个 worker 都有 apps
✅ 本地 JOIN
👉 这是你最该用的模式

3-最坑的 JOIN（跨 shard JOIN） ​
sql
orders (order_id)
users  (user_id)

SELECT *
FROM orders o
JOIN users u ON o.user_id = u.id;
1
2
3
4
5
6

问题

  • 分片键不同
  • JOIN 条件 ≠ 分片键

后果

  • Coordinator 拉全量数据
  • 或禁止执行（新版本更严格）
  • 性能灾难

📌 这是为什么“分布式库设计 > SQL 技巧”

JOIN 限制背后的底层原因 ​

Citus ​不能随便在节点间 shuffle 数据（不像 Spark）：

  • PostgreSQL 执行器是单机模型
  • 跨节点数据移动 = 巨大成本
  • 所以 设计阶段就要求你选好分片键
分片键怎么选？（结合你实际业务） ​

黄金原则

JOIN 最频繁的字段 = 分片键

sql
单机 PG 分区
   ↓
Citus 分布式表
   ↓
每个 shard 内再做时间分区（可选）

部署 ​

一、Coordinator 要怎么部署？ ​
1️⃣ 本质一句话 ​

Coordinator = 一台“几乎不存数据”的 PostgreSQL + Citus

它不是新东西，就是一个 ​PG 实例，只是：

  • 开了 citus 扩展
  • 扮演调度角色
2️⃣ Coordinator 的最小部署步骤（裸机 / 云主机） ​

① 安装 PostgreSQL（版本要和 worker 一致）

例如 PG 15：

bash
sudo apt install postgresql-15
1

② 安装 Citus 扩展

bash
sudo apt install postgresql-15-citus
1

③ 配置 postgresql.conf（核心）

conf
shared_preload_libraries = 'citus'
listen_addresses = '*'
1
2

📌 必须重启

④ 创建数据库 & 启用扩展

sql
CREATE DATABASE demo;
\c demo
CREATE EXTENSION citus;
1
2
3

到这一步，​Coordinator 已经就位。

3️⃣ Coordinator 机器的“角色定位建议” ​
项建议
存数据❌ 不建议
CPU中等即可
内存偏大（SQL 规划 / 结果合并）
磁盘不重要
网络很重要

👉 Coordinator = 控制节点，不是算力节点

二、Worker 要怎么部署？ ​
1️⃣ Worker 的本质 ​

Worker = 一台“标准 PostgreSQL + Citus”，真正存数据 & 执行 SQL

注意一句话：

Coordinator 和 Worker 在软件层面几乎一模一样

不同的是：

  • Worker 会被“登记”到 Coordinator
  • 数据会分布到 Worker
2️⃣ Worker 部署步骤（和 Coordinator 几乎一样） ​

① 安装 PG + Citus

bash
sudo apt install postgresql-15 postgresql-15-citus
1

② postgresql.conf（重点）

conf
shared_preload_libraries = 'citus'
listen_addresses = '*'
1
2

③ pg_hba.conf（非常重要）

允许 Coordinator IP 访问 Worker：

conf
host  all  all  <COORDINATOR_IP>/32  md5
1

📌 这是 worker 能“被调度”的前提

④ 重启 PostgreSQL

bash
sudo systemctl restart postgresql
1
3️⃣ Worker 不需要做的事（新手常误解） ​

❌ 不需要：

  • 手动建分布式表
  • 手动建 shard
  • 连接应用
  • 配任何“主从”

👉 Worker 完全被 Coordinator 控制

三、Worker 是怎么和 Coordinator 通信的？（重点） ​
1️⃣ 通信本质：PostgreSQL 普通连接 ​

没有 RPC、没有 gRPC、没有消息队列

就是：

text
Coordinator
   |
   |  libpq (TCP)
   |
Worker
1
2
3
4
5

使用的协议：

  • PostgreSQL wire protocol
  • TCP 5432
2️⃣ 通信是谁主动的？ ​

✅ 永远是 Coordinator 主动

Client → Coordinator
Coordinator → Worker
1
2

📌 Worker 不会主动找 Coordinator

3️⃣ Coordinator 如何“认识” Worker？ ​

通过这条命令 👇

sql
SELECT citus_add_node('worker_ip', 5432);
1

这一步做了什么？

  • 在 Coordinator 元数据表里记录：

    • worker IP
    • 端口

  • 之后所有路由都基于这个信息

4️⃣ SQL 执行时的真实通信流程（你一定要看） ​

查询：

sql
SELECT count(*)
FROM system_log
WHERE tenant_id = 3;
1
2
3

内部流程：

1. Coordinator 解析 SQL
2. 找到 tenant_id = 3 → shard_102015
3. shard_102015 在 Worker B
4. Coordinator 建立 PG 连接
5. 发送子 SQL
6. Worker 执行
7. 返回结果
1
2
3
4
5
6
7

📌 这就是“计算分散”的核心

5️⃣ INSERT 的通信流程（很轻） ​
sql
INSERT INTO system_log VALUES (3, 'xxx');
1
Coordinator → Worker（1 次网络）
Worker → Coordinator（OK）
1
2
四、部署时你最容易踩的 5 个坑（非常实战） ​
❌ 1️⃣ Coordinator 和 Worker 版本不一致 ​
  • PG / Citus 版本必须一致
❌ 2️⃣ pg_hba.conf 没放行 ​
  • 这是最常见错误
  • 表现为：add_node 成功，执行 SQL 失败
❌ 3️⃣ Coordinator 放了大量数据 ​
  • JOIN / 查询压力会反噬 coordinator
❌ 4️⃣ 网络延迟高 ​
  • Citus 对网络非常敏感
  • 不要跨机房
❌ 5️⃣ 把 Worker 暴露给业务 ​
  • 应用只连 coordinator
五、给你一个“部署后自检清单” ​

在 Coordinator 执行：

sql
SELECT * FROM citus_get_active_worker_nodes();
1

再执行：

sql
EXPLAIN SELECT count(*) FROM system_log;
1

看到：

Custom Scan (Citus Adaptive)
1

👉 说明通信 & 调度全部 OK

六、工程级一句话总结（记住这句） ​

Coordinator 不干活，只指挥；Worker 不思考，只执行；它们用的只是最普通的 PostgreSQL 连接。

⚠️ 下面步骤 4 台机器都要做（包括 coordinator）

1️⃣ 安装 PostgreSQL + Citus ​

以 Ubuntu 22.04 + PG15 为例：

四、Worker 节点需要做什么？ ​

👉 答案：几乎什么都不用做

Worker：

  • 不用建库（citus 会同步）
  • 不用建表
  • 不用连应用

只要：

  • PostgreSQL 在跑
  • 能被 coordinator 访问

你可以在任意 worker 上验证：

bash
sudo -u postgres psql

sql
SELECT citus_is_coordinator();

👉 返回 f，说明它是 worker

六、常见错误 & 对应排查（你很可能会遇到） ​
❌ add_node 时报错连接失败 ​

👉 99% 是：

  • pg_hba.conf 没放行
  • 防火墙没开 5432

检查：

bash
telnet 192.168.1.113 5432
1
❌ EXPLAIN 没有 Citus Adaptive ​

👉 表不是 distributed table

sql
SELECT * FROM citus_tables;
1
❌ 插入很慢 ​

👉 分片键没写 / 值不均匀

七、非常重要的生产建议（先记住） ​
  • ❌ 不要让业务连 worker
  • ❌ 不要在 worker 手动建表
  • ✅ coordinator 独立一台
  • ✅ 所有机器时间同步（NTP）
