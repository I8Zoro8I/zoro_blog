pg_trgm —— 模糊搜索 / like / 搜索性能质变 ​

1. 它解决什么问题？ ​

你现在是不是经常写这种 SQL 👇：

sql
WHERE title ILIKE '%虾%'
   OR desc ILIKE '%虾%'
1
2

问题是：

  • %xxx% 必定全表扫描
  • 数据一多直接炸

pg_trgm 能让 模糊搜索走索引。

2. 安装插件（只需一次） ​
sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
1

验证：

sql
SELECT * FROM pg_extension WHERE extname = 'pg_trgm';
1
3. 给字段建 trigram 索引（核心） ​
sql
CREATE INDEX idx_recipe_title_trgm
ON recipe
USING gin (title gin_trgm_ops);
1
2
3

如果你经常：

sql
title + desc 一起搜
1

可以建 组合索引：

sql
CREATE INDEX idx_recipe_search_trgm
ON recipe
USING gin ((title || ' ' || desc) gin_trgm_ops);
1
2
3

💡 这个在 菜谱 / 日志 / 内容搜索里非常好用。

4. 实测效果（你一定要做） ​
sql
EXPLAIN ANALYZE
SELECT *
FROM recipe
WHERE title ILIKE '%大虾%';
1

3
4

你会看到：

  • ❌ 没插件 → Seq Scan
  • ✅ 有插件 → Bitmap Index Scan / Index Scan

这一步一定要亲眼看，你才会真的理解插件的价值。

5. 你现在项目里可以立刻用的地方 ​

结合你已有的模块 👇

  • 菜谱搜索（title / desc）
  • 商品名称搜索
  • 日志 message 模糊匹配
  • 用户备注、操作说明

全文检索 tsvector ​

tsvector 是 PostgreSQL 内置类型，只要数据库版本 ≥ 8.3（你现在用的肯定远高于这个），开箱即用。

  • tsvector：不用安装
  • ✅ to_tsvector() / to_tsquery()：直接可用
  • ❌ 只有 分词规则 / 扩展能力 才可能需要额外插件

6.自动维护 ​

1-触发器

sql
CREATE FUNCTION system_log_tsvector_trigger()
RETURNS trigger AS $$
BEGIN
  NEW.search_vec :=
      setweight(to_tsvector('simple', coalesce(NEW.level,'')), 'A')
  ||  setweight(to_tsvector('simple', coalesce(NEW.module,'')), 'B')
  ||  setweight(to_tsvector('simple', coalesce(NEW.message,'')), 'C');
  RETURN NEW;
END
$$ LANGUAGE plpgsql;
CREATE TRIGGER trg_system_log_tsv
BEFORE INSERT OR UPDATE
ON system_log
FOR EACH ROW
EXECUTE FUNCTION system_log_tsvector_trigger();

表分区 ​

1.为什么日志一定要分区？ ​

结合你现在 & 未来的场景：

  • 日志只会 追加
  • 查询 99% 带时间条件
  • 旧日志几乎不查，但要保留

👉 时间分区 = 天然完美解

2..推荐分区策略 ​
sql
PARTITION BY RANGE (log_time)
1

为什么不是按月？

  • 日志量不稳定
  • 单分区过大 → 索引臃肿
  • 日级最灵活（删、归档、冷热分离）

2-自动创建"未来N天分区" ​

命名规范 xxx_log_YYYYMMDD

sql
system_log_20251224
1

创建分区函数

sql
CREATE OR REPLACE FUNCTION create_system_log_partition(p_date date)--'2025-12-24'
RETURNS void AS $$
DECLARE
    v_start timestamptz := p_date;
    v_end   timestamptz := p_date + interval '1 day';
    v_table text := format('system_log_%s', to_char(p_date, 'YYYYMMDD'));
BEGIN
--原生 declarative partition
    EXECUTE format(
        'CREATE TABLE IF NOT EXISTS %I
         PARTITION OF system_log
         FOR VALUES FROM (%L) TO (%L)',
        v_table, v_start, v_end
    );

    -- 时间索引
    EXECUTE format(
        'CREATE INDEX IF NOT EXISTS %I ON %I (log_time)',
        v_table || '_log_time_idx', v_table
    );

    -- 全文检索索引
    EXECUTE format(
        'CREATE INDEX IF NOT EXISTS %I
         ON %I USING gin (search_vec)',
        v_table || '_search_idx', v_table
    );
END;
$$ LANGUAGE plpgsql;
--一次性建立未来7天
DO $$
DECLARE
    d date;
BEGIN
    FOR d IN CURRENT_DATE .. CURRENT_DATE + 7 LOOP
        PERFORM create_system_log_partition(d);
    END LOOP;
END $$;

3-自动删除过期分区 ​

删除函数

sql
CREATE OR REPLACE FUNCTION drop_old_system_log_partition(p_keep_days int)
RETURNS void AS $$
DECLARE
    v_cutoff date := CURRENT_DATE - p_keep_days;
    r record;
BEGIN
    FOR r IN
        SELECT inhrelid::regclass AS part
        FROM pg_inherits
        WHERE inhparent = 'system_log'::regclass
    LOOP
        IF substring(r.part::text FROM '(\d{8})')::date < v_cutoff THEN
            EXECUTE format('DROP TABLE IF EXISTS %s', r.part);
        END IF;
    END LOOP;
END;
$$ LANGUAGE plpgsql;
